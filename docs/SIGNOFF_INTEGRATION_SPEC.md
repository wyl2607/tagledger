# Sign-Off Integration Spec

## Goal

Define a sanitized integration plan for using TagLedger as the local-first capture and matching workspace for return sign-off work, with an assistive PC browser module for an external RMA SaaS.

The intended workflow is:

- Phone operators capture product labels, return labels, and supporting photos.
- TagLedger extracts business identifiers through OCR and barcode recognition.
- The backend matches phone-side captures to return sign-off tasks by business identifiers.
- A PC-side assistant helps an operator prepare fields on the external RMA SaaS return sign-off page.
- The operator reviews every prepared action and performs the final save or submit manually.

This document is an engineering planning contract. It must not preserve exploration logs, tenant details, account details, local paths, browser session artifacts, page-locator details, screenshots, or exact third-party operating instructions.

## Non-goals

- Do not automate final submission to the external RMA SaaS in the first implementation slice.
- Do not store external SaaS credentials, browser session files, tenant names, or private network details in the repository.
- Do not document exact third-party page-locator details, page URLs, DOM structures, or reproducible reconnaissance steps.
- Do not make the external RMA SaaS the source of truth for TagLedger inventory, OCR history, or operator audit trails.
- Do not replace TagLedger's local-first factory workbench, mobile capture flow, outbound workflow, or admin roles.

## User Workflow

1. A phone operator opens the TagLedger mobile capture flow on the local factory network.
2. The operator selects or scans the active return context, then captures photos of the machine label, return label, and any required supporting evidence.
3. TagLedger runs OCR and barcode recognition, then proposes normalized identifiers such as product model, serial number, return authorization number, or shipment reference.
4. The operator reviews and confirms the extracted values in TagLedger.
5. The backend groups confirmed captures into a return sign-off candidate using business identifiers and capture metadata.
6. A PC operator opens the external RMA SaaS in a normal browser session and navigates to the relevant return sign-off page.
7. The PC-side assistant receives only the sanitized candidate payload needed to prepare the page.
8. The assistant presents a dry-run preview of the fields and attachments it would prepare.
9. The operator compares the preview against the external RMA SaaS page and confirms any local preparation step.
10. The operator performs the final save or submit action manually inside the external RMA SaaS.

## Data Model Draft

The first slice can use a minimal local model that stays independent from the third-party system internals.

### Return Sign-Off Candidate

- `id`: local TagLedger identifier.
- `status`: local workflow status, such as `draft`, `ready_for_assist`, `assist_previewed`, `manually_completed`, or `needs_review`.
- `business_key`: normalized matching key derived from operator-confirmed identifiers.
- `return_reference`: optional operator-confirmed return reference.
- `product_model`: optional normalized model value.
- `serial_number`: optional normalized serial value.
- `captured_at`: local capture timestamp.
- `confirmed_by`: local TagLedger operator identifier.
- `evidence_photo_ids`: local references to captured images managed by TagLedger.
- `ocr_confidence_summary`: coarse confidence metadata for review prioritization.
- `notes`: local operator notes.

### Assist Session

- `id`: local TagLedger identifier.
- `candidate_id`: linked return sign-off candidate.
- `pairing_key_id`: reference to the active pairing key.
- `mode`: must be `dry_run` for the first implementation slice.
- `prepared_payload_hash`: checksum of the sanitized payload shown to the operator.
- `previewed_at`: timestamp of the latest assist preview.
- `operator_decision`: local decision, such as `accepted_for_manual_entry`, `rejected`, or `needs_correction`.
- `external_completion_mark`: optional human-entered note that the operator completed the SaaS action manually.

### Evidence Photo

- `id`: local TagLedger identifier.
- `candidate_id`: linked return sign-off candidate.
- `type`: controlled value such as `machine_label`, `return_label`, `package`, or `other`.
- `storage_ref`: internal TagLedger storage reference, not an absolute machine path.
- `capture_device`: coarse device category, such as `phone`.
- `ocr_text_summary`: sanitized OCR summary for local review.

## Pairing Key Concept

The pairing key links a phone-confirmed return sign-off candidate to a PC-side assist session without embedding external SaaS credentials or browser session data.

- A pairing key is generated by TagLedger for one candidate or one short-lived batch.
- The key is scoped to local TagLedger state and expires quickly.
- The key authorizes the PC-side assistant to fetch only the sanitized candidate payload.
- The key does not grant permission to perform final SaaS submission.
- The key must not contain customer credentials, tenant identifiers, network addresses, or browser storage material.
- The PC assistant should display the candidate identity and payload hash so the operator can verify that the phone-side capture and PC-side page correspond to the same return.

## Phone-Side Responsibilities

- Capture photos needed for return sign-off evidence.
- Run or trigger OCR and barcode extraction through TagLedger.
- Let the operator confirm or correct extracted business identifiers.
- Attach capture metadata needed for local traceability.
- Submit confirmed captures to the backend as local TagLedger records.
- Avoid interacting directly with the external RMA SaaS.
- Avoid storing or displaying external SaaS credentials, browser sessions, or tenant-specific page details.

## Backend Responsibilities

- Persist confirmed phone-side captures and evidence references.
- Normalize business identifiers used to match return sign-off candidates.
- Create and update local return sign-off candidates.
- Issue short-lived pairing keys for PC assist sessions.
- Serve sanitized candidate payloads to paired PC assistants.
- Record assist previews, operator decisions, and manual completion notes.
- Keep `dry_run` or assistive mode as the only enabled mode for the first implementation slice.
- Enforce that no backend endpoint performs the final external SaaS save or submit action in the first implementation slice.
- Keep external SaaS configuration out of committed source unless it is placeholder-only and non-operational.

## PC-Side Assistant Responsibilities

- Run as an operator-controlled helper for a browser already opened by the PC operator.
- Fetch only the sanitized candidate payload authorized by the pairing key.
- Present a clear dry-run preview before any page preparation.
- Prepare fields or attachments only after local operator confirmation.
- Stop before the final save or submit control.
- Show that final completion must happen manually in the external RMA SaaS.
- Record local assist outcome back to TagLedger without claiming third-party completion unless the operator marks it manually.
- Avoid logging page-locator details, page URLs, tenant details, browser session state, or copied third-party page content.

## Security And Privacy Boundaries

- Repository documentation must use abstract terms such as external RMA SaaS, tenant login, and return sign-off page.
- Committed files must not include real accounts, email addresses, tenant names, internal network addresses, local machine paths, browser session artifacts, precise URLs, screenshots, or page-locator details.
- Local runtime artifacts must stay outside Git according to the repository sync rules.
- Pairing keys must be short-lived, scoped, and revocable.
- Evidence photos remain TagLedger-managed local records until an operator manually chooses how to use them.
- The first slice must be safe to test without contacting the external RMA SaaS or mutating third-party state.
- Any future real-submit capability requires a separate design review, test plan, credential handling review, and explicit operator confirmation model.

## Manual Confirmation Rule

All final external RMA SaaS save, sign-off, completion, or submit actions must be performed by a human operator.

The PC-side assistant may prepare a draft, focus a field, stage an attachment, or display a checklist in assistive mode. It must not click or trigger the final external action in the first implementation slice. TagLedger may record that an operator marked a candidate as manually completed, but that local mark is not proof of third-party completion unless supported by a separate approved audit design.

## Open Questions

- Which local business key should be the primary match signal when OCR finds multiple plausible identifiers?
- Should one pairing key authorize a single candidate, a short batch, or both?
- What evidence photo types are required for the first return sign-off pilot?
- What local role is allowed to create pairing keys and mark candidates as manually completed?
- How should TagLedger handle a candidate that was prepared on PC but rejected by the operator after visual comparison?
- What minimal audit fields are needed without storing third-party page content or sensitive browser data?

## Acceptance Criteria For First Implementation Slice

- Phone capture can create or update a local return sign-off candidate from operator-confirmed OCR or barcode values.
- The backend can issue a short-lived pairing key for one candidate.
- A PC-side assistant can fetch a sanitized candidate payload with that pairing key.
- The assistant can show a dry-run preview of intended field and evidence preparation.
- The assistant cannot trigger final external RMA SaaS save, sign-off, completion, or submit actions.
- Operators can mark a candidate as manually completed or needs review inside TagLedger.
- Logs and committed files contain no real SaaS credentials, account details, tenant identifiers, internal network addresses, local machine paths, browser session artifacts, precise third-party URLs, screenshots, or page-locator details.
- The slice can be validated without contacting or mutating the external RMA SaaS.
